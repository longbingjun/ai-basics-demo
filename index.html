How to Use AI · 基础认知篇

业务人员如何理解 AI

按章节逐步理解：先掌握核心概念，再看模型从训练到输出的链条，用翻译例子串起概率生成机制，最后了解从简单对话到项目化使用的演进路径。

Token

模型读写文字时的最小单位，来自一张固定的「词表」，不等于一个字或一个词。

划分依据：词表里的条目，是训练前从海量文本中统计高频片段得到的。切分时从左到右，优先匹配词表中最长的片段——常见词/短语整段保留，少见词拆成更小的片段。

英文

Hello world → 常见词，通常整段匹配：

Hello world

uncharacteristically → 少见长词，词表里没有整词，拆成子片段：

uncharacteristically
中文

我想要学习英语 → 同一规则，高频片段优先：

我想要学习英语

「想要」「英语」在语料中出现频率高，容易成为独立 token；不是按新华字典的「字」来切。

Prompt

你发给模型的完整任务说明，告诉它做什么、怎么做、不要做什么。

日常对比
不清楚 帮我写个生日祝福。
清楚 给我妈写一段 80 字以内的生日祝福，语气温馨不要太正式，不要提到年龄。

Prompt 不是「魔法咒语」，而是把需求说清楚。说得越具体，模型越知道该输出什么风格和内容。

上下文

模型在当前这一轮对话里能「看见」的全部信息，且有长度上限。

日常构成

你这次的问题 + 之前的聊天记录 + 系统设定（如「你是一个翻译助手」）。内容越长，占用的 token 越多；超过上限时，最早的内容会被挤掉。

推理（概率生成）

大模型是概率输出模型：根据当前已有的 token，逐个预测下一个最可能出现的 token，直到生成结束。

翻译时的内部过程（简化）

模型已经输出了 I，接下来要在词表里选下一个 token。在预训练和后训练见过的语料中，I 后面跟 want 的情况非常多，所以 want 的预测概率最高，模型就选择输出 want。然后基于 I want 继续预测下一个，如此循环，直到输出句号。

因此模型不是在「查标准答案库」，而是在做概率最高的连续选择——这就是为什么同一个问题可能有多种合理回答。

幻觉

模型用流畅、自信的语言，输出了错误、编造或缺乏依据的内容。

为什么会发生？

① 模型的本职是「预测看起来合理的下一段文字」，不是「核实事实是否正确」——即使上下文里没有可靠信息，它仍会输出统计上最像答案的内容。

② 训练数据本身可能有错误、过时或互相矛盾的信息，模型学到的是「文本中的规律」，不是「经过验证的真值」。③ 当被问到具体细节（人名、日期、数据）而模型没有足够依据时，它倾向于「补全」一个听起来合理的说法，而不是回答「我不知道」。

日常例子

问 AI「某小众餐厅昨天有没有营业」——模型可能编造一个具体答案，因为它在预测「这类问题后面通常跟着怎样的回答」，而不是去查营业记录。

RAG（检索增强生成）

Retrieval-Augmented Generation：在模型生成回答之前，先从外部资料库中检索相关内容，再把检索结果放进 Prompt，让模型「带着依据」来回答。

实现过程的本质（四步）：

① 用户提问 → ② 系统从文档库中检索与问题最相关的段落 → ③ 把检索到的段落插入 Prompt（作为上下文的一部分）→ ④ 大模型基于这些段落生成回答。

与大模型的关系：RAG 不替代大模型，也不改变它的概率生成机制。大模型仍然是一个 token 一个 token 地预测输出；RAG 做的是在生成前给它「开卷资料」，让预测有材料可依，而不是只靠模型内部的「记忆」。

与幻觉的关系：当问题的答案确实存在于资料库中，且检索到了正确段落时，模型有据可查，幻觉会明显减少。但如果资料库没有相关内容、检索不准、或资料本身有误，模型仍可能编造——RAG 是降低幻觉的手段，不能彻底消除幻觉。

补充：「工具调用」是让模型能读文件、运行程序、访问外部系统；RAG 解决「依据从哪来」，工具调用解决「手能不能伸出去做事」——两者常配合使用。

出厂前：模型怎么被做出来

数据收集 收集海量文本作为模型的「阅读材料」 网页、书籍、代码、问答、翻译对照等。材料越多样，模型见过的语言场景越广。
预训练 Next Token Prediction：给定前面的 token，猜下一个 token 在海量文本上重复这一任务亿万次。模型逐渐学会语法、常识和知识之间的统计关联——例如「I」后面常跟「want」，「学习」和「learn」常出现在类似位置。
后训练 在「会续写」的基础上，学会听指令、守格式 用指令-回答样例教模型：看到「请翻译」就执行翻译，而不是随意续写；看到「不要添加额外含义」就遵守约束。本质是调整概率偏好，让输出更符合人类期望。

预训练和您使用时的推理，底层机制相同——都是 next token prediction。区别是：预训练读的是整个互联网，推理时读的是您的 Prompt 和上下文。

使用时：你提问后发生什么

用户输入 把任务、规则和要求打包成 Prompt Prompt 被切成 token 序列，连同历史对话一起进入模型的上下文窗口。
推理生成 在您的上下文后面，逐个 token 做概率预测并输出 每一步都在所有候选 token 中选概率最高的（或按策略采样）。直到生成结束标记或达到长度限制。
人工复核 模型输出的是「概率上合理」的文本，不是「已验收的结论」 涉及事实、数字、合规的内容，必须有人确认。这也是幻觉存在的根本原因——模型不会自行核实。

Step 1 · 你写了什么（Prompt）

用户希望 AI 把一句中文翻译成自然英文，并且不要添加额外含义。

Prompt
请把「我想要学习英语」翻译成自然英文，不要添加额外含义。
期望输出
I want to learn English.

这就是一份任务说明书：做什么（翻译）、翻译什么（「我想要学习英语」）、约束是什么（自然英文、不加额外含义）。

Step 2 · 模型先「读」（切分与编码）

整句 Prompt 被切成 token，模型先读完所有输入，再开始生成。

Prompt 中的中文部分
我想要学习英语

切分依据与第一部分相同：按词表匹配高频片段。读入完成后，模型对整句任务形成内部表示——可以理解为「它看全了再动笔写」。

Step 3 · 模型再「写」（逐 token 概率生成）

模型在 Prompt 后面，一个 token 一个 token 地预测输出。每一步都在候选词中选概率最高的：

已生成的内容 预测的下一个 token 为什么概率高
（空，准备开始翻译） I 翻译「我（想要）……」的英文句，常以 I 开头
I want 预训练语料中 I 后接 want 频率极高；「想要」→ want 的对应关系统计上最常见
I want to want 后接 to 是英文高频搭配
I want to learn 「学习」在类似语境中常对应 learn
I want to learn English 「英语」→ English 的直接对应
I want to learn English . 完整句子，句号结束

这就是第一部分的「推理（概率生成）」在实际任务中的完整展开——不是查词典，而是逐步做概率最高的选择。

Step 4 · 这些能力从哪来？为什么不是唯一答案？

预训练提供了中英文之间的统计规律（I want to learn English 是高频模式）。后训练让模型看到「请翻译……不要添加额外含义」时，执行翻译任务而非乱续写。

为什么有多种合理英文？因为每一步都在多个候选中选「概率较高」的，不是查唯一标准答案。例如 I would like to learn English 也合理（更礼貌），I want to study English 也通顺（study 更偏「攻读」）。若模型在某一步选了不同的高概率 token，整句就会变化——这再次说明需要人工复核，也可以在 Prompt 里进一步约束风格。

下面是从「聊一句」到「做成套」的演进路径——不是一开始就要全用上，而是任务变复杂、变重复时，逐步引入对应能力。

1
起点

聊天式使用

在对话框里问一句、答一句，靠 Prompt 临时说明需求。适合探索性、一次性的任务。

2
加材料

带上下文的使用

上传文件或多轮补充背景，让模型「看到更多」。适合分析一份材料、基于特定内容回答。

3
写清楚

Documents（文档）

给人和 AI 共同阅读的说明文件，把规则、术语、输出格式写进固定文档，不用每次在聊天里重复粘贴。

术语表.md · 输出格式说明.md · 流程指引.txt
4
有依据

Reference（参考依据）

AI 执行任务时必须查的权威来源——通常是上面的文档加上业务规则、字段定义。对应第一部分的 RAG：把资料检索出来再让模型回答，减少幻觉。

5
能重复

Workflow（工作流）

多步骤、常重复、顺序固定的做事流程。适合每周都要跑的同类任务，用文档或工具固化步骤。

6
能稳定

Script（脚本）

规则完全确定、不需要 AI「发挥」的步骤，写成可执行代码。

process.py · run.ps1 · Excel 公式
7
可复用

Skill（技能包）

把「何时触发 + 用什么文档 + 什么步骤 + 什么工具」打包，下次直接复用。

SKILL.md + 样例 + 模板
8
能动手

Agent（智能体）

大模型 + 工具 + 权限 + 执行循环：能自己规划步骤、读文件、跑脚本、迭代修正。适合步骤不确定、需要多轮操作的复杂任务。

9
可回退

Git（版本管理）

记录文档、脚本、规则改了什么，方便协作、审查和回退。多人维护同一套规则或脚本时使用。

1. 说清目标明确你要 AI 做什么，而不是只说「帮我看看」。
2. 给足上下文提供规则、文件、样例、术语表和限制条件。
3. 写明边界说明哪些不要擅自判断，哪些必须人工确认。
4. 先计划再执行复杂任务先让 AI 说明步骤，再授权它动手。
5. 复核并沉淀重要结果要检查，重复任务要变成 Workflow 或 Skill。
